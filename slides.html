Exploratory Data Analysis

The first step for any data science project is to exploratory data analysis (or EDA). Exploring and understanding your data is crucial for your work to be effective. If you build a model based on incorrect assumptions about your data, the model will be flawed and you might waste a lot of time trying to figure out why. So the goal in EDA is to gain a thorough understanding of your data and locate and resolve any potential issues with your data.

Data formats

Let's start with the data. There are a number of different formats your data can be in. These include the ever-present excel spreadsheet, a csv (comma-separated values) file, a tsv (tab-separated values) file, JSON, and more.
Here's what a csv file would look like. I used the built-in linux command head to print out the first 10 lines of the file.
The format of your data might depend on its source. There are many different types of databases. One common type is a relational database, which focuses on the relationships between the data. Examples include mySQL, SQL Server, MS Access, PostgreSQL. This type of database requires normalization, which means that different related tables are joined (linked) so that there is no redundancy. While this helps with data integrity, it can make the database requests slow. An alternative would be a noSQL database, such as MongoDB, Redis, Apache Cassandra, HBase. They do not require normalized data and therefore are known for being very fast and scalable. A lot of these can be distributed, meaning that the data is spread across multiple machines. This would allow you to store lots and lots of data and be able to query very quickly.

Data Issues

  • Typos
  • Missing data
  • Redundant data
  • Formatting
  • Outliers
So basically with all the plots and summary stats, we're trying to understand what the data looks like and discover any possible issues. These issues could include typos, missing data, redundant data, formatting issues, and outliers. When you see these issues, it should be a red flag, to talk to your data provider about, so you can decide how you want to handle them. While you're exploring your data, you may notice some issues. There may be typos, missing data, redundant data, format issues, or outliers. When you notice these issues, the first step is to discuss with data provider. You want to know things like how the data was collected, what exactly it is (sometimes the column names are misleading), if any of the data is stale or deprecated (sometimes one column was deprecated, but left in database, and replaced by a more accurate column). This information will help you decide how to handle the various issues with the data.

The biggest difference I see between the communities is that statistics emphasizes inference, whereas machine learning emphasized prediction. When you do statistics, you want to infer the process by which data you have was generated. When you do machine learning, you want to know how you can predict what future data will look like w.r.t. some variable. What is A/B testing?


VS.
So what is A/B testing? A/B testing is an experiementation technique where you show some customers one thing (version A) and other customers another (version B). It's commonly used in web analytics, where a company would show some percentage of the visitors to its website one layout or color scheme or ad, and others a different one. You typically want to keep the rest of the content the same, so that you can attribute the difference in click-throughs to the the change you're testing.

Multi-Armed Bandit

One common situation that's related to A/B testing is the multi-armed bandit problem. The name comes from the idea of a slot machine or "one-armed bandit" having two arms to pull. You may have some hypothesis of which arm would have the bigger reward, but you need to come up with a strategy that effectively tests your theory and also maximizes the reward. So in terms of A/B testing in online ads, you would want to choose the perecentage of website visitors to show each ad that provides enough data to give confidence to your findings but also maximize click-throughs in the process. There are a number of different strategies you can take here. The simpliest is called epsilon-greedy, which assigns a fixed percentage (say 90%) of the visitors to the ad that is expected to have the higher reward. But in order to gain more information about the other ad, you may consider a probability matching strategy, such as the Bayesian bandit strategy. You can use Bayes Theorem to come up with a prior probability, that's your starting point. And it updates based on each new data point.

Multivariate Testing


VS.

VS.

VS.

An alternative to A/B testing is multivariate testing. As the name suggests, multivariate testing tests multiple variables at the same time. You can use a full factorial method, which would test every combination of the changes, or a partial factorial method, which would only test a fraction of the combinations and infer the rest. While you can learn more about the effectiveness of each variable, the testing is more complex and requires more data (visitors) than A/B testing, which is sometimes not worth the effort.

What NOT to do


No peeking!
One of the most important things to keep in mind when A/B testing is when to stop. Generally, you want to stop testing when you reach a certain level of statistical significance. However, this should be calculated beforehand. Often people "peek" at their current significance, and stop when they reach a certain level. However, this might be influenced by a current streak, for example, so it would be misleading. So, you should calculate your sample size, based on your desired level of significance, beforehand and do not stop early.
